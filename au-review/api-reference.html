Trillet API Reference

Complete reference for the Trillet voice AI platform API. Validated against live production, 31 March 2026.
Base URL: https://api.trillet.ai
Versions: v1 (most resources), v2 (calls, user)

Introduction

The Trillet API lets you build and manage voice AI agents programmatically: create agents, configure call flows, provision phone numbers, initiate calls, and retrieve transcripts and analytics. The API is a REST API. All requests and responses use JSON.

Two versions run in parallel. Most resources use /v1/api/. Call history, active calls, recordings, and user profile use /v2/api/. Where the version matters, it is noted on each endpoint.

Documentation accuracy: The official docs at docs.trillet.ai contain significant path errors. Several documented endpoints use incorrect paths, wrong API versions, or describe features that do not exist via the REST API. This reference reflects the live production API only.

Authentication

Every request needs two headers. There are no optional auth methods.

HeaderValueDescription
x-api-key Your API key Generated at app.trillet.ai > Settings > Developer > API Keys. Required on every request
x-workspace-id Your workspace ID The MongoDB ObjectId of your workspace. Found in your JWT access_token cookie payload. Required on every request
Content-Type application/json Required on all POST and PUT requests 
# Every curl command needs these headers
curl -H "x-api-key: YOUR_API_KEY" \
     -H "x-workspace-id: YOUR_WORKSPACE_ID" \
     -H "Content-Type: application/json" \
     "https://api.trillet.ai/v1/api/agents"

JWT parity: JWT Bearer tokens and API keys hit identical routes. There is no separate JWT-only API surface. Any endpoint returning 404 with an API key returns 404 with a JWT.

API Versions

VersionUsed for
/v1/api/Agents, call flows, knowledge base, phone numbers, workspace, crews, flow folders, messaging, outbound calls, batch calls
/v2/api/Call history, active calls, recordings, user profile
Common mistake: The official docs often show v1 paths for call history and recordings. These endpoints are v2 only. Using v1 returns 404.

Errors

StatusMeaningCommon causes
400Bad requestMissing required field, invalid enum value, malformed JSON
401UnauthorizedMissing or invalid x-api-key
403ForbiddenValid key but insufficient permissions for this workspace
404Not foundRoute not registered, or resource ID does not exist
409ConflictDuplicate resource (e.g., phone number already in workspace)
500Server errorPlatform bug, invalid ID format, missing service method. See Bug Report for known 500s

All error responses include a message field in the response body describing the issue.

Silent failures: Several known bugs cause the API to return 200 when the operation did not actually succeed. Always verify writes with a follow-up GET. See Behavioral Notes.

Behavioral Notes

PUT replaces, it does not merge

All PUT endpoints use replace semantics. Sending a partial object causes omitted fields to revert to defaults or empty values. The API returns 200 with your submitted values, making it look like a successful partial update. A GET afterward shows the omitted fields are gone.

Always follow this pattern for updates:

# 1. GET the full current state
CURRENT=$(curl -s "https://api.trillet.ai/v1/api/agents/$ID" -H "x-api-key: ...")

# 2. Merge your change into the full object
UPDATED=$(echo "$CURRENT" | python -c "
import json, sys
d = json.load(sys.stdin)
d['name'] = 'New Name'  # only change what you intend to change
print(json.dumps(d))
")

# 3. PUT the complete merged object
curl -X PUT "https://api.trillet.ai/v1/api/agents/$ID" \
  -H "x-api-key: ..." -H "Content-Type: application/json" \
  -d "$UPDATED"

Call flow ownership

Call flow PUT requests only persist on flows that were created or duplicated by the same API key user. PUT a flow owned by a different workspace user and you get 200, but nothing changes. No error is returned. Always verify writes with a GET.

Draft and publish

The Trillet UI saves call flow changes as drafts. Publishing requires POST to /call-flows/{id}/drafts/{draftId}/publish. API PUTs bypass drafts and go directly to the live configuration. If a team uses both the UI and the API, be aware that UI drafts and API writes are separate paths to the same live config.

Verification protocol

For any operation that creates or modifies a resource, always verify the result with a follow-up GET before proceeding to the next step. Compare the fields you sent against what came back. Do not trust a 200 response alone.

Agents

Agents define the AI voice model for calls: which LLM, which TTS voice, which STT provider, and behavioral settings like speed, temperature, and smart callback. Each agent is associated with one call flow (via the pathway field) and one or more phone numbers.

Endpoints

POST
/v1/api/agents
Create agent

Request body

FieldTypeRequiredDescription
namestringRequiredDisplay name for the agent
llmModelstringRequiredLLM model ID. See Models reference for full list
ttsModelobjectOptional{provider, voiceId, language}. See Voices reference
sttProviderobjectOptional{id: "deepgram-nova-3" | "deepgram-flux" | "whisper"}
settingsobjectOptional{speed: float, volume: float, temperature: float}
smartCallbackbooleanOptionalEnable smart callback retry logic for missed calls
maxAttemptsForSmartCallbackintegerOptionalMax retry attempts (default 3)
gapBetweenEachAttemptForSmartCallbackarrayOptionalArray of {days, hours} intervals between retries
pathwaystringOptionalCall flow ID to associate with this agent
ttsDictionaryobjectOptionalPronunciation dictionary config
fallbackSettingsobjectOptionalFallback speed/volume/temperature if primary fails

Example request

curl -X POST "https://api.trillet.ai/v1/api/agents" \
  -H "x-api-key: YOUR_KEY" \
  -H "x-workspace-id: YOUR_WORKSPACE" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Sales Assistant",
    "llmModel": "gpt-4.1",
    "ttsModel": {
      "provider": "rime",
      "voiceId": "mistv2_Brook",
      "language": "en"
    },
    "sttProvider": {"id": "deepgram-nova-3"},
    "settings": {"speed": 1.0, "volume": 1.0, "temperature": 0.7},
    "smartCallback": true
  }'

Response

200 with full agent object including _id, status ("ready"), createdAt.

GET
/v1/api/agents
List agents

Returns all agents in the workspace. No pagination. Each agent includes full config, phoneNumberIds array, and pathway.

GET
/v1/api/agents/{id}
Get agent

Returns the full agent object for the given ID, including all voice config, settings, phoneNumberIds, pathway, and activeSessions.

PUT
/v1/api/agents/{id}
Update agent

Updates the agent. Replace semantics: GET the full agent first, merge your changes, then PUT the complete object. Partial objects wipe omitted fields.

Same body structure as POST. All fields optional on update.

DELETE
/v1/api/agents/{id}
Delete agent

Deletes the agent. Returns 200 with the full deleted agent object (not 204).

POST
/v1/api/agents/{agentId}/bind/{phoneNumberId}
Bind phone to agent

Sets the phone number's pathwayId to the agent's associated flow and flowType to "call". This is what makes inbound calls route to the agent. Without this step, inbound calls to the number get a disconnected tone.

Request body

{
  "recordCalls": false,
  "recordingPublic": false,
  "officeHours": {"enabled": false},
  "metadataTags": {},
  "label": "",
  "transferCallerIdMode": "from-transferee"
}
Note: Set recordCalls: true here to enable call recording. Recording cannot be toggled mid-call via API.
GET
/v1/api/agents/voices
List available voices

Returns available TTS voices grouped by provider.

Response structure

{
  "openai": [/* 6 voices */],
  "elevenlabs": [/* 127 voices */],
  "hume": [/* 102 voices */]
}
Note: Rime and Cartesia voices are not returned by this endpoint. Set them directly via voiceId string in the agent's ttsModel config.

Call Flows

Call flows define the agent's behavior during a call: the system prompt, welcome message, call settings (max duration, silence handling, office hours), transfers, post-call actions, and integrations. Each flow has 36 configurable settings fields, most of which are undocumented in the official docs.

Known bug: Flow creation via POST is broken. POST /v1/api/call-flows returns 500 on all payloads. The workaround is to duplicate an existing flow via POST /v1/api/call-flows/{id}/duplicate, then configure it with PUT.

Endpoints

POST
/v1/api/call-flows/{id}/duplicate
Duplicate flow (primary creation method)

Creates a copy of the specified flow owned by the API key's user. Use this instead of the broken POST /call-flows endpoint.

Request body

{}

Returns the new flow with a new _id. The name will be "<original> (Copy)". Configure the new flow using PUT.

GET
/v1/api/call-flows
List call flows

Returns all call flows in the workspace. Includes full config: prompt, nodes, edges, all settings, promptVersions array.

GET
/v1/api/call-flows/{id}
Get call flow

Full flow detail including promptVersions history. Use currentPromptId to identify the active prompt version.

PUT
/v1/api/call-flows/{id}
Update call flow

Updates the call flow configuration. Replace semantics. GET the full flow first, merge your changes, PUT the complete object.

Ownership required: PUT only persists on flows you own (created or duplicated with your API key). PUT a non-owned flow gets 200, but nothing is written. No error is returned.

Top-level fields

welcomeMessagestring"user_initiated", "ai_dynamic", or "ai_custom"
customWelcomeMessagestringUsed when welcomeMessage is "ai_custom" for bidirectional flows
agentstringAgent ID to associate with this flow
settingsobject36-field settings object. See Flow Settings Reference

/v1/api/call-flows/{id}
Delete call flow

Returns 204 No Content.

POST
/v1/api/call-flows/{id}/drafts/{draftId}/publish
Publish draft to live

Equivalent to the "Push Live" button in the UI. The UI saves changes as drafts first. Call this endpoint to push a draft to the live configuration. List drafts via GET /v1/api/call-flows/{id}/drafts to get the draftId.

API PUTs bypass the draft system entirely and write directly to live.

Outbound Calls

Initiate AI-driven outbound calls to any phone number.

POST
/v1/api/call
Initiate outbound call
FieldTypeRequiredDescription
tostringRequiredDestination phone number (E.164)
call_agent_idstringRequiredAgent ID to handle the call
callback_urlstringOptionalWebhook URL for call status updates
dynamic_variablesobjectOptionalKey-value pairs injected into prompt as {{variable_name}}
metadataobjectOptionalPassed through to webhooks. Not included in agent context

Prerequisites

  1. Agent must have at least one phone number assigned via the bind endpoint
  2. Agent's pathway field must point to an active call flow
  3. If office hours are enabled: call must be placed within the configured window, or calls queue until the next open slot
Office hours: Calls placed outside the configured hours sit in pending status with scheduledTime set to the next available window. Disable office hours in the call flow settings to call on demand at any time.

Web Calls

Start and end browser-based voice calls using WebRTC via LiveKit. Used for web widgets, demos, and embedded voice experiences. Requires the agent's call flow to have public access enabled.

POST
/v1/api/calls/web/connect
Start web call
FieldTypeRequiredDescription
agentIdstringRequiredAgent ID. Must have an active pathway configured

Response

{
  "status": "success",
  "callId": "...",
  "roomName": "...",
  "token": "...",       // LiveKit JWT for WebRTC connection
  "agent": {"id": "...", "name": "...", "model": "...", "voice": "..."}
}

Use the roomName to poll active call endpoints during the call. Use the token with the LiveKit JS SDK or Trillet Web SDK to connect WebRTC audio.

/v1/api/calls/web/disconnect
End web call
{"callId": "...", "roomName": "..."}

Returns {"status": "success", "message": "Session ended successfully"}.

SMS Configuration

Post-call SMS is configured in the call flow's settings.smsPostCallSettings object. Three fields are all required for SMS delivery to work.

PUT
/v1/api/call-flows/{id}
Configure post-call SMS (via flow settings)

Include this object in the flow's settings on a full-object PUT:

{
  "smsPostCallSettings": {
    "businessOwnerSMS": true,           // send SMS to business owner after each call
    "customerSMS": false,               // send SMS to caller after call
    "useInternalA2P": true,             // REQUIRED: enable the A2P number pool
    "selectedPhoneNumberId": "",        // empty = auto-select by region
    "businessOwnerPhoneNumber": "+1XXXXXXXXXX"  // REQUIRED: exact field name
  }
}
Silent failure: Missing useInternalA2P: true or using the wrong field name (businessOwnerPhone instead of businessOwnerPhoneNumber) causes SMS to silently not send. No error is returned.

A2P SMS costs USD $0.008/SMS surcharge in addition to call costs.
