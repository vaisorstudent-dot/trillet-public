Documentation Issues Report

All known discrepancies between docs.trillet.ai and live production behavior

This report documents every known problem with the official Trillet documentation at docs.trillet.ai, based on seven rounds of direct API testing between 29 and 31 March 2026. Issues fall into four categories: wrong API paths, wrong API versions, undocumented endpoints, and behavioral gaps. Eight confirmed platform bugs are covered separately in the API Bug Report.

Synopsis & Commercial Ramifications

The official Trillet documentation is not functional as a standalone integration resource. Less than half the API surface is correctly documented. Nine endpoints required for basic operations are absent from the docs entirely. None of the seven behavioral patterns that cause silent failures are documented anywhere. A developer following only docs.trillet.ai cannot complete a single end-to-end task: inbound call routing, phone number provisioning, and call history retrieval all require undocumented or incorrectly documented endpoints.

  • 1
    Agency activation is blocked. The two endpoints critical to deploying an inbound agent (bind phone to agent, purchase phone number) are completely absent from the docs. An agency developer starting from scratch will fail before writing a single line of production code. This directly impairs the agency channel that Barak and Ori are building.
  • 2
    Silent data loss goes undetected and unattributed. The undocumented replace semantics on PUT and the flow ownership model mean developers configure agents that appear to work, then lose configuration silently. Without knowing to run a GET after every PUT, this failure mode is nearly impossible to debug. Support tickets will describe it as "settings resetting" or "configuration not saving" with no path to resolution.
  • 3
    Support ticket volume is inflated by documentation-preventable failures. Wrong paths produce 404s. Wrong version prefixes produce 404s. These look like platform bugs to the developer. Each one generates a support contact. The current support load almost certainly contains a significant proportion of documentation-originating tickets, particularly around call history (wrong version) and phone number management (wrong path).
  • 4
    Enterprise evaluation fails before it starts. Enterprise procurement teams evaluating the API will typically spend 1-2 hours testing against the documentation before engaging sales. At 41% documentation accuracy, they will hit 404s and silent failures within minutes. The experience reads as an unstable or immature API, which is a hard perception to reverse even when the underlying platform works correctly.
  • 5
    The office hours queuing trap creates invisible outbound call failures. Developers building outbound campaigns will place calls that enter pending status with no explanation. Calls appear to be accepted (200 response) but never connect. Without knowing about the office hours queuing behavior, developers will spend hours ruling out agent configuration, phone number issues, and network problems before discovering the real cause.
What it takes to fix it: The 9 undocumented endpoints and 7 behavioral notes can be documented in a single targeted update. The 5 wrong paths and 6 wrong versions are find-and-replace corrections. Total documentation effort is estimated at 2-3 days for a technical writer with live API access. Priority order: (1) bind endpoint + phone provisioning endpoints (unblock agency activation), (2) correct v1/v2 version mismatches, (3) behavioral notes section covering PUT semantics, flow ownership, and SMS requirements.

Category 4: Behavioral Gaps

These behaviors are not mentioned or explained anywhere in the official documentation. The API performs them consistently. Without knowing about them, developers encounter silent failures and data loss with no diagnostic path.

1. PUT uses replace semantics, not merge semantics

Every PUT endpoint replaces the entire stored object with the submitted payload. Omitted fields revert to defaults or become empty. The API returns 200 with your submitted values, so the write looks successful. A GET afterward reveals the omitted fields are gone. This affects agents, call flows, knowledge bases, workspaces, and every other mutable resource. Not documented anywhere.

Fix: GET the full resource first, merge changes into the full object, PUT the complete merged object back.

2. Call flow ownership model for PUT

PUT on a call flow only persists if the flow was created or duplicated by the same API key user. Sending a PUT to a flow owned by a different workspace user returns 200 with the submitted values echoed back, but a subsequent GET shows the original values unchanged. No error is returned at any point. Not documented anywhere.

3. Draft and publish model

Changes made through the web UI auto-save to a draft rather than going live immediately. Going live requires clicking "Push Live" (API: POST /call-flows/{id}/drafts/{draftId}/publish). API PUTs bypass the draft system entirely and write directly to live. The two write paths target different states and are not interchangeable. Not explained in the documentation.

4. Office hours queue outbound calls rather than blocking them

When office hours are configured and an outbound call is placed outside those hours, the call enters pending status with a scheduledTime set to the next open window. The API accepts the call (200 response) but does not connect it. No error is returned. Developers expecting on-demand calling will see calls stuck in pending with no explanation. Not documented anywhere.

5. SMS requires three specific fields

Post-call SMS requires all three of: businessOwnerSMS: true, useInternalA2P: true, and businessOwnerPhoneNumber (exact field name matters — businessOwnerPhone is accepted by PUT but may not be read by the SMS pipeline). Missing any one results in silent failure: the setting appears saved, no error is returned, no SMS is delivered. None of these requirements are documented.

6. GET responses may return stale data immediately after PUT

A GET request immediately following a successful PUT may return the previous values rather than the updated ones due to caching. For call flows, the most reliable freshness indicator is the promptVersions array: a new entry appears when a PUT successfully persisted. For other resources, a 1-2 second wait before re-GET resolves the issue.

7. Transfer schema: old field names accepted but not used at runtime

The transfer object in call flow settings accepts two sets of field names: the old names (label, phoneNumber) and the current names (friendly_name, destination). Old names are accepted by PUT and echo back on GET — they appear saved. At runtime, the transfer_call function only reads the current names. Transfers configured with old field names will fail with "Team not found" on every attempt. Not documented anywhere.
