API Capability Matrix

Programmatic Access Reference for Enterprise Integration Teams

Trillet has a REST API for managing voice agents, call flows, phone numbers, and calls. This document maps every major platform feature to one of three categories: accessible via API, UI-only, or currently broken at the server level. All findings come from seven rounds of live testing against api.trillet.ai between 29 and 31 March 2026, using API key auth, JWT Bearer auth, Playwright network capture during browser sessions, and active voice call tests. Two API versions run in production. Most resources use /v1/api/. Call history, active calls, and recordings use /v2/api/.

Note on official documentation: The docs at docs.trillet.ai use incorrect paths for a significant number of endpoints. Where the documented path differs from the working path, the correct path is shown in this document. The discrepancies are detailed in the full API reference.

Key Behavioral Notes

PUT replaces, it does not merge
Send a partial object to any PUT endpoint and all omitted fields revert to defaults or empty values. The API returns 200 with your submitted values, so the write looks successful. A GET afterward shows the omitted fields are gone. Always GET the full resource first, merge your changes in, then PUT the complete object back. This applies to agents, call flows, knowledge bases, and everything else.
Call flow ownership
PUT only persists on flows you created or duplicated with your API key. PUT a flow owned by another workspace user and you get 200 back, but nothing changes. No error is returned. The API claims success. When building automations that modify call flows, always ensure the flow was created or duplicated by the same API key user performing the update.
Draft and publish
The web UI saves call flow changes to a draft first. Going live requires clicking "Push Live," which triggers POST /call-flows/{id}/drafts/{draftId}/publish. API PUTs skip the draft system entirely and write directly to the live configuration. If a team uses both the UI and the API to manage the same flows, be aware that UI drafts and API writes operate as separate paths to the live config.
Office hours and outbound calls
If office hours are enabled on a call flow, outbound calls placed outside the configured window are queued rather than placed immediately. They sit in pending status until the next open slot. The scheduledTime field on the call record shows when it will be placed. To call on demand without scheduling constraints, either disable office hours or confirm calls are being placed within the configured window.
SMS: three required fields
Post-call SMS via the A2P pool requires three specific fields in smsPostCallSettings: businessOwnerSMS: true, useInternalA2P: true, and businessOwnerPhoneNumber (the exact field name the SMS pipeline reads; businessOwnerPhone is accepted by PUT but not read by the SMS service). Missing any one of these causes silent failure with no error returned.
Transfer configuration schema
The transfers array in flow settings accepts two field formats, but the transfer_call function only reads the UI format. Using the older API format (label, phoneNumber) causes the agent to report "Team not found" at transfer time despite the transfer appearing correctly in settings. Always use: friendly_name, destination (E.164), method ("cold"/"warm"), destinationType ("SIP").

Authentication Reference

Header Value Notes
x-api-key Your API key Generated at app.trillet.ai > Settings > Developer > API Keys
x-workspace-id Your workspace ID Required on every request. Found in the workspace URL or JWT access_token payload
Content-Type application/json Required on POST and PUT requests
JWT vs API key: JWT Bearer tokens and API keys hit identical routes. There is no separate JWT-only API surface. Any endpoint returning 404 with an API key will return 404 with a JWT. Features that appear UI-only are implemented in Next.js server-side routes, not in the public REST API.
