Confirmed API Bug Report

Live Production Issues for Developer Team Review

This report documents eight confirmed bugs in the Trillet REST API, each with a reproducible test case, the exact server response, and a working workaround where one exists. All bugs were discovered during seven rounds of live testing between 29 and 31 March 2026. Testing used API key auth, JWT Bearer auth, Playwright network capture during browser sessions, and real-time calls. The environment was the live production API at api.trillet.ai.

Detailed Bug Reports

BUG-01
Call flow creation via POST always returns 500
High
Endpoint
POST /v1/api/call-flows
Expected
New call flow object with 201 or 200 status
Actual
500 on every payload variant tested: minimal (name+direction only), with agentId, with full config, bidirectional and outbound
Reproduction
curl -X POST "https://api.trillet.ai/v1/api/call-flows" \
  -H "x-api-key: YOUR_KEY" \
  -H "x-workspace-id: YOUR_WORKSPACE" \
  -H "Content-Type: application/json" \
  -d '{"name": "Test Flow", "direction": "bidirectional"}'
# Returns: 500
Impact
Full flow provisioning automation is blocked. Can not create new flows programmatically
Workaround
Duplicate an existing flow: POST /v1/api/call-flows/{sourceId}/duplicate with body {}. This returns a new flow owned by the API key user. Configure it via PUT.
First confirmed
29 March 2026
BUG-02
PUT on non-owned call flows: silent write failure, false 200
High
Endpoint
PUT /v1/api/call-flows/{id} (when flow is owned by a different workspace user)
Expected
403 Forbidden or 400 with "not authorized to modify this flow"
Actual
200 with the submitted payload echoed back. Subsequent GET returns original values unchanged. No error at any point
Reproduction
1. Obtain a call flow ID owned by any user other than the API key's user.
2. PUT a config change to that flow ID.
3. GET the same flow ID.
4. Observe that the GET response reflects original values, not the submitted update.
Impact
Silent data integrity failure. Automations that modify shared flows report success, but changes are never applied. Extremely difficult to debug without explicit verification after every write
Workaround
Only PUT flows created or duplicated by the same API key user. Verify every write with a GET
First confirmed
29 March 2026
BUG-03
Replace semantics on all PUT endpoints: partial objects wipe fields silently
Medium
Affected endpoints
All PUT endpoints: /v1/api/agents/{id}, /v1/api/call-flows/{id}, /v1/api/knowledgebase/{id}, and all other mutable resources
Expected
Merge semantics: only the fields sent in the body are updated; all other fields are unchanged
Actual
Replace semantics: the entire object is replaced with the submitted payload. Omitted fields revert to defaults or empty values. The API returns 200 with the submitted values, making the write appear successful
Example
An agent has a configured TTS voice, STT provider, and smart callback settings. Send PUT with only {"name": "Updated Name"} and all other fields revert to defaults. The agent's voice, STT, and callback config are gone.
Note
This behavior may be intentional (some APIs use replace semantics by design). If intentional, the documentation should state it explicitly. The Trillet docs currently imply standard update behavior with no mention of full-object replacement
Workaround
GET the full resource first. Merge changes into the complete object. PUT the entire merged object. Confirm that only the intended fields changed with a follow-up GET.
First confirmed
29 March 2026
BUG-04
Voice search route registered but service method not implemented
Medium
Endpoint
GET /v1/api/voices/search
Expected
Search results from the voice catalog
Actual
500: "voiceService.searchVoices is not a function"
Reproduction
curl "https://api.trillet.ai/v1/api/voices/search?query=female" \
  -H "x-api-key: YOUR_KEY" \
  -H "x-workspace-id: YOUR_WORKSPACE"
# Returns: 500 {"message": "voiceService.searchVoices is not a function"}
Additional note
Also tested with JWT auth. Same result. The route is registered in the router but the underlying service class does not implement the searchVoices method
Workaround
Use GET /v1/api/agents/voices for the full voice catalog (returns OpenAI, ElevenLabs, Hume voices grouped by provider)
First confirmed
31 March 2026
BUG-05
Workspace delete fails with type cast error
Medium
Endpoint
DELETE /v1/api/workspaces/{id}
Expected
Workspace deleted (204 or 200)
Actual
500: "Cast to string failed for value ... at path 'deletedBy'"
Cause
The deletedBy field in the workspace schema expects a string user ID. The API key auth context provides a different user object format than the handler expects, causing a Mongoose cast error
Workaround
Delete workspaces via the Trillet web UI
First confirmed
29 March 2026
BUG-06
On-demand SMS generation: returns success but does not deliver
High
Endpoint
POST /v1/api/conversations/message/generate
Expected
AI-generated SMS delivered to the target number. Dynamic variables interpolated into the message template
Actual
API returns {"status": "success", "conversation_id": "...", "message_id": "..."}. No SMS is received by the target number. Dynamic variables passed in the dynamic object are not interpolated into the initiationMessage template.
Reproduction
curl -X POST "https://api.trillet.ai/v1/api/conversations/message/generate" \
  -H "x-api-key: YOUR_KEY" \
  -H "x-workspace-id: YOUR_WORKSPACE" \
  -H "Content-Type: application/json" \
  -d '{
    "flowId": "YOUR_MESSAGE_FLOW_ID",
    "to": "+1XXXXXXXXXX",
    "metadata": {"autopilot": true},
    "dynamic": {"customer_name": "Test"}
  }'
# Returns: {"status": "success"} -- but no SMS delivered
Additional note
Post-call SMS via smsPostCallSettings on call flows works correctly and does deliver. This bug affects only the on-demand SMS generation path
Workaround
Use smsPostCallSettings on the call flow for automated post-call SMS. Requires useInternalA2P: true and businessOwnerPhoneNumber fields
First confirmed
30 March 2026
BUG-07
Webhook settings silently drop on call flow PUT
Medium
Endpoint
PUT /v1/api/call-flows/{id} with settings.webhooks
Expected
Webhook URLs persist and are returned on subsequent GET
Actual
PUT returns 200 with webhook values in the response body. GET the same flow and settings.webhooks is empty. Values never persisted despite the 200
Reproduction
curl -X PUT "https://api.trillet.ai/v1/api/call-flows/FLOW_ID" \
  -H "x-api-key: YOUR_KEY" -H "x-workspace-id: YOUR_WORKSPACE" \
  -H "Content-Type: application/json" \
  -d '{"settings": {"webhooks": {"inbound": "https://example.com/hook"}}}'
# Returns 200 with webhooks in body

curl "https://api.trillet.ai/v1/api/call-flows/FLOW_ID" \
  -H "x-api-key: YOUR_KEY" -H "x-workspace-id: YOUR_WORKSPACE"
# settings.webhooks is {} -- values never saved
Workaround
Configure webhooks via the flow editor in the Trillet web UI (Data > System Webhooks tab)
First confirmed
29 March 2026
BUG-08
Remove-agent (human takeover) fails on web calls with LiveKit encoding error
Medium
Endpoint
POST /v2/api/active-calls/{roomName}/remove-agent
Required body
{"participantIdentity": "<agent identity from participants array>"}
Expected
Agent is removed from the call, allowing a human operator to take over
Actual
500: "cannot encode field livekit.SendDataRequest.kind" when called during a web call session
Scope
Confirmed only on web calls (WebRTC via LiveKit). The endpoint is not yet tested on standard telephony calls (inbound/outbound PSTN). The error suggests a web-call-specific data encoding issue in the LiveKit integration
Body note
Without the participantIdentity field, the server returns 400 "Participant identity is required." The field is validated correctly. The error occurs after validation during the LiveKit call
Workaround
None confirmed for web calls. For telephony: needs testing. End the web call via POST /v1/api/calls/web/disconnect if human takeover is required
First confirmed
31 March 2026

Testing Methodology

All bugs in this report were discovered and confirmed through direct API testing against the live production environment. No staging or sandbox environment was available. Testing covered:

  • Direct REST calls via curl and Python, with both API key and JWT Bearer authentication
  • Playwright browser network capture during live UI sessions, used to discover undocumented endpoints and compare UI vs API behavior
  • Active voice call testing, including a live web call with real-time transcript and insight polling
  • Seven distinct validation passes between 29 and 31 March 2026
JWT parity confirmed: JWT Bearer tokens and API keys hit identical routes on the Trillet API. All bugs confirmed via API key were re-tested with JWT and produced the same results. There is no separate JWT-only API surface that resolves any of these issues.
