Tier & Pricing
D2C Channel Conflict Resolved

Is D2C at $49 the same product agencies white-label at $300-$500?

Ming (Mar 12-13): D2C at $49 is Trillet-branded, not white-labeled. Agencies get the same product as a template plus 4 vertical templates, all white-labeled under their own brand. Separation is structural at the brand level.

5 agency templates today: D2C equivalent (general), Receptionist, Legal, Debt Collection, Recruitment.

Ming's honest nuance (Mar 13 call): The product IS the same. Agencies WILL lose some customers who discover the $49 price. Two customer segments: (1) self-sufficient users who leave when they find direct option, (2) users who can't self-serve ("how do I even connect it to my phone number?") who happily pay $300/mo. D2C is a wedge for new agencies to get first clients (GHL review-service analogy). Agencies evolve away from D2C as they learn to build custom automation. Fixer analogy: $30/mo email automation replaced $6-10K agency builds -- complexity is the moat.

Revenue model (Ming, Mar 13 calls 2-3): No rev share for standard agency tiers. Definitive. "We got to take our base and be done. We're gonna make money on usage on the agency tiering model." Biggest competitors don't do rev share. The historical 60/40 profit share is going away. BUT: custom enterprise deals CAN do subsidized build fee + full rev share (Quebec model). Two enterprise paths: (a) one-time build fee $12-35K, (b) subsidized build with rev share.

"Powered by" separation (Ming, Mar 13): D2C SMS shows "powered by [brand name]". White-label agencies get different SMS protocol without "powered by." Agency owners confirmed no problem with D2C existing this way.

Resolution: Insighto model with accepted leakage. Brand-level separation is real, channel conflict is also real. Mitigation is customer capability segmentation, not product differentiation. Footer-only D2C placement limits discovery surface. Revenue model locked: no rev share standard, rev share enterprise-only.

API Surface High

What does the API currently expose, and can it be split?

The tier table claims "Operational API" at every tier. Mar 9 audit found ~90 endpoints. API is flat (no tier gating). "Templates" do not exist as an API concept (product has flows). Agent config identical for all users. Integrations ungated.

Proposed split: operational API (read, manage agents, webhooks, usage data) open at all tiers. Commercial API (plan creation, pricing changes, automated signup flows, billing enforcement, provisioning) at Scale only. Does the current endpoint surface support this distinction?

Ming (Mar 12, partial): Rate limits exist on how many agents can be deployed via API. This gating needs to be restricted to Agency Scale tier only. Implementation work required to enforce the tier restriction.

What it affects: Whether API gating is enforceable with current architecture or needs engineering. Also determines if "Operational API at Entry" is a real claim. Rate limit gating at Scale gives a concrete enforcement mechanism.

Product-Surface Grounding High

What is the complete consumer-facing feature list?

We are in an open tier model: same product at every tier, no feature gating. To position and market this correctly, we need the definitive list of what a customer actually gets. Not internal capabilities or backend architecture, but what a user sees and can do.

For each feature: what is it called, what does it do, and is it live today? Categories: agent building (voice, prompts, KB, integrations, call settings), white-label (branding, custom domain, powered-by removal), billing and markup (client billing, subscription creation, usage metering), workspace management (sub-accounts, team roles, permissions), and API (what endpoints are available).

Partial answer (Ming, Mar 13 calls 2-3): Two distinct creation flows confirmed: (1) 5-minute builder -- generates dynamic questions based on how AI understood the business, generic; (2) Vertical templates -- SET intake question sets with baked-in domain expertise ("we know how lawyers and law firms should respond"), curated. Custom agents: Scale ($497) only, definitive. Also: "Cruise" is a new multi-agent orchestration feature for hallucination prevention at enterprise scale (e.g., verified customer context persists across conversation turns). Not API-able, enterprise only.

What it affects: The website feature list, pricing page comparison table, and sales collateral. We cannot market features we cannot name precisely. This is the foundation for the agency and enterprise pages. 5-min builder vs template distinction affects how we present agent creation UX.

Agent Config Object High

What is the agent config object today?

When you create an agent, what is the full atomic config? Prompt, voice, KB, integrations, call settings, speech behavior. What fields exist, what is structured vs freeform?

Why it matters: This is the foundation. The parameterization engine can only parameterize what the config object exposes. If the config is unstructured freeform, full-stack parameterization requires restructuring first.

Parameterization Resolved

How hard is full-stack parameterization?

Ming (Mar 12): Trillet has full template capability through Level 3 (Template) of the ontology. Parameterized templates with intake forms are live. This is not aspirational -- "fill form, get working agent" exists today.

Level 4 (Deployment Pack) individual components exist but are not strung together as a unified package. This is an assembly job, not a build. Comms Channel validation (Mar 13): Comms Channel used all of Trillet's API tools and wired them into a 1-click deploy -- they "wired it together because they know their system." This is live L4 proof: a partner independently assembled a deployment pack from Trillet components. Level 5 (Marketplace Asset) remains future, viable at 400-500 agencies.

Template depth example -- Legal (Mar 13 call): Case classification, PMS integration, multi-lawyer routing with separate phone numbers and calendars per lawyer. Not just prompts -- "prompt plus business know-how plus interfaces." Each vertical template has domain-specific integration logic pre-configured.

Builder vs template distinction (Ming, Mar 13): 5-min builder = dynamic AI-generated questions, generic. Templates = curated intake with SET question sets and baked-in domain expertise. These are two different products serving different customer capability levels.

Implication: Trillet already has what no competitor offers -- full-stack parameterization in voice AI. ChatDash only parameterizes prompt text. GHL parameterizes sub-accounts but is not voice AI. Trillet does both. Ming committed to spot-checking ChatDash/Voice AI Wrapper "template" claims -- believes wrappers can't change parameters inside Retell/Vapi API.

BYOK Resolved

BYOK / BYO-provider at Scale: do agencies want it?

Ming (Mar 12-13): BYOK is live and available on the platform today. Not a future build item. Operational necessity: Trillet's current ElevenLabs plan has 110-voice limit, can't bring full library. Customers want full catalog access. BYOK bridges this gap.

Remaining positioning question: how prominently to feature BYOK in tier architecture. Competitors (Assistable, Insighto, VoiceAIWrapper) all offer it. Trillet already has it. The 110-voice limit adds a practical reason beyond positioning: agencies with voice-sensitive clients need BYOK for full catalog.
